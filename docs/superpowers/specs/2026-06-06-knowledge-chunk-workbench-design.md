# 知识分块工作台设计

## 1. 背景与目标

当前知识详情页同时展示文件列表和分块列表，文件管理空间不足，分块也只能查看，无法直接维护。目标是将页面拆分为两个职责明确的层级：

- 知识详情页负责文件管理。
- 单击文件后进入分块工作台，负责浏览、搜索、修改和重新向量化单个分块。

本期只实现单分块编辑、名称维护和单分块重新向量化。不包含新增、删除、合并、拆分、拖拽排序或批量编辑分块。

## 2. 已确认的产品决策

- 分块页采用大纲式工作台。
- 左侧为紧凑分块大纲，右侧为固定编辑面板。
- 编辑面板支持“编辑 / 预览”切换。
- 保存正文后自动异步重新向量化当前分块。
- 向量化失败后由 RocketMQ 自动重试，最终失败时允许用户手动重试。
- 分块名称由 AI 初次生成，同时允许用户手动修改。
- 用户手动修改名称后锁定，后续正文修改不自动覆盖；用户可主动执行“AI 重新生成”。
- 同步期间允许继续编辑，通过内容版本号保证仅最新版本生效。
- 不使用或改造现有 `agent_knowledge_job` 表、调度器和本地事件机制。

## 3. 页面信息架构

### 3.1 文件列表页

文件列表页使用全宽紧凑表格，优先服务文件管理。单击文件行进入该文件的分块工作台，不在文件列表页常驻展示分块区域。

文件表格建议展示：

- 文件名称
- 解析状态
- 切分策略
- 分块数量
- 文件大小
- 更新时间
- 操作

### 3.2 分块工作台

工作台由顶部栏、左侧大纲和右侧编辑区组成。

#### 顶部栏

- 返回文件列表
- 文件名称
- 分块总数
- 已同步、处理中、失败数量

#### 左侧大纲

固定宽度约 `280px`，使用紧凑单行条目：

`#12  分块名称  同步状态`

支持：

- 按分块名称和正文搜索
- 按全部、处理中、失败筛选
- 当前分块高亮
- 键盘上下切换分块
- 始终展示处理中和失败状态

大纲不展示大段正文摘要，避免挤占浏览空间。分块名称过长时省略，并通过提示展示完整名称。

#### 右侧编辑区

- 顶部展示可编辑分块名称、名称锁定状态和“AI 重新生成”操作。
- 中部通过标签切换 Markdown 编辑和渲染预览。
- 底部固定操作栏展示字符数、内容版本、向量版本、同步状态和保存按钮。
- 存在未保存修改时，切换分块或离开页面需要确认。
- 保存后不阻塞继续编辑，并持续展示最新版本的异步同步状态。

## 4. 分块数据模型

在 `agent_knowledge_chunk` 增加以下字段：

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `name` | `VARCHAR(255)` | AI 生成或用户维护的分块名称 |
| `name_locked` | `TINYINT` | `0` 表示允许 AI 自动更新，`1` 表示用户已锁定 |
| `content_version` | `INT` | 当前正文版本，初始值为 `1` |
| `vector_version` | `INT` | 当前已成功向量化的正文版本，未成功时为空 |
| `vector_status` | `VARCHAR(32)` | 当前最新正文版本的同步状态 |
| `retry_count` | `INT` | 当前版本的消费重试次数 |

沿用现有 `error_msg` 字段记录最终失败原因。现有 `status` 字段在迁移后逐步由语义更明确的 `vector_status` 替代，避免一个字段同时表达切分状态与向量同步状态。

`vector_status` 状态定义：

- `PENDING`：数据库内容已保存，等待消费。
- `PROCESSING`：正在生成新向量。
- `SYNCED`：当前 `content_version` 已成功向量化。
- `FAILED`：RocketMQ 重试耗尽，等待手动重试。

## 5. 名称生成规则

### 5.1 初次切分

文件切分并保存分块后，为每个分块生成简短名称。名称生成失败时使用正文首个非空行作为降级名称；仍为空时使用“分块 #{chunkOrder}”。

名称生成不阻塞分块向量化。初次生成的名称设置 `name_locked = 0`。

### 5.2 编辑正文

- 当 `name_locked = 0` 时，正文保存成功后异步重新生成名称。
- 当 `name_locked = 1` 时，正文保存不会修改名称。
- 用户手动修改名称时设置 `name_locked = 1`。
- 用户点击“AI 重新生成”后，使用当前最新正文生成名称，并将 `name_locked` 设置为 `0`。

名称生成结果写入前必须再次校验 `content_version`，并确认 `name_locked = 0`。旧版本生成结果或用户手动锁定名称后晚到的 AI 结果均不得覆盖当前名称。

## 6. 单分块保存与版本并发

### 6.1 保存请求

前端提交：

- `name`
- `content`
- `contentVersion`
- 名称是否由用户手动修改

后端使用 `chunk_id + content_version` 进行乐观锁更新：

1. 校验分块属于当前 `agentId` 和 `knowledgeId`。
2. 更新正文、正文长度、名称和名称锁定状态。
3. 将 `content_version` 加一。
4. 设置 `vector_status = PENDING`、`retry_count = 0`，清空 `error_msg`。
5. 数据库事务提交后发送 RocketMQ 消息。
6. 返回新版本和当前同步状态。

若版本冲突，接口返回冲突错误，不覆盖数据库内容。前端保留本地编辑内容，并提示用户重新加载后处理冲突。

### 6.2 连续编辑

用户可在向量化期间继续保存。每次保存产生新的 `content_version` 和对应消息。

消费者处理消息前后都必须校验消息版本是否仍为当前 `content_version`。旧版本消息直接忽略，不得覆盖最新状态、名称或向量版本。

## 7. RocketMQ 分块消息设计

RocketMQ 只用于分块相关异步能力，不依赖现有任务表、调度器和本地事件。

### 7.1 配置

基础配置使用用户提供的参数，并允许通过环境变量覆盖：

```yaml
rocketmq:
  name-server: ${ROCKETMQ_NAME_SERVER:118.195.146.161:9876}
  producer:
    group: ${ROCKETMQ_PRODUCER_GROUP:data-agent-producer-group}
    send-message-timeout: 5000
    retry-times-when-send-failed: 2
```

建议补充：

- Topic：`data-agent-knowledge-chunk`
- 向量化消费者组：`data-agent-knowledge-chunk-vector-consumer`
- 名称生成消费者组：`data-agent-knowledge-chunk-name-consumer`
- Tag：`VECTORIZE`、`GENERATE_NAME`

### 7.2 消息内容

消息携带最小定位与并发信息：

```json
{
  "agentId": 1,
  "knowledgeId": 10,
  "chunkId": "knowledge-10-chunk-3",
  "contentVersion": 4
}
```

消费者始终从数据库读取当前内容，不直接信任消息中的正文或名称。

### 7.3 发送可靠性

保存事务提交后发送消息。若发送失败：

- 保存接口返回“内容已保存，但同步任务提交失败”的明确结果。
- 分块保持 `PENDING` 状态。
- 用户可在页面执行手动重试，再次发送当前最新版本消息。

本期不引入 Outbox、任务表或后台补偿扫描，因此必须明确接受“发送失败后依赖用户重试”的边界。

### 7.4 向量化消费者

消费者按以下顺序处理：

1. 查询分块并校验消息版本仍为最新版本。
2. 将最新版本状态更新为 `PROCESSING`。
3. 使用版本化向量 ID，例如 `{chunkId}-v{contentVersion}`，写入新向量。
4. 再次校验版本仍为最新版本。
5. 将 `vector_version` 更新为消息版本，状态更新为 `SYNCED`。
6. 删除该分块除当前版本外的旧向量。

采用“先写新向量、成功后切换并清理旧向量”，避免失败期间分块完全无法被检索。

消息重复消费时，版本化向量 ID 和状态校验保证幂等。旧版本消息晚到时直接忽略，也不得删除新版本向量。

### 7.5 失败与重试

- 短暂异常抛回 RocketMQ，由 Broker 触发重试。
- 每次消费失败更新当前版本的 `retry_count` 和 `error_msg`。
- 消息进入死信队列后，由死信消费者将仍为当前版本的分块标记为 `FAILED`。
- 若失败消息对应旧版本，只记录日志，不修改分块最新状态。
- 用户点击“重新同步”时，将当前最新版本状态重置为 `PENDING`，清空错误与重试次数，并重新发送消息。

## 8. API 设计

### 8.1 分块大纲分页查询

`GET /api/v1/agent-knowledge/{knowledgeId}/chunks`

请求参数：

- `agentId`
- `keyword`
- `vectorStatus`
- `page`
- `pageSize`

返回名称、顺序、字符数、版本、同步状态和更新时间，不返回完整正文。

### 8.2 获取分块详情

`GET /api/v1/agent-knowledge/{knowledgeId}/chunks/{chunkId}`

返回完整正文、名称、名称锁定状态、版本、同步状态、重试次数和错误信息。

### 8.3 保存分块

`PUT /api/v1/agent-knowledge/{knowledgeId}/chunks/{chunkId}`

请求体包含名称、正文、当前 `contentVersion` 和名称手动修改标记。接口返回更新后的分块详情以及消息是否成功提交。

### 8.4 手动重新同步

`POST /api/v1/agent-knowledge/{knowledgeId}/chunks/{chunkId}/retry`

仅允许针对当前最新版本重新发送向量化消息。

### 8.5 AI 重新生成名称

`POST /api/v1/agent-knowledge/{knowledgeId}/chunks/{chunkId}/generate-name`

发送当前最新版本的名称生成消息，并解除名称锁定。接口不阻塞等待 AI 返回。

## 9. 前端状态与反馈

- 保存按钮状态：`保存`、`保存中`、`已保存`。
- 向量状态：`等待同步`、`同步中`、`已同步`、`同步失败`。
- 保存接口受理后不得直接显示“已同步”。
- 当 `contentVersion > vectorVersion` 时，始终展示“最新修改等待同步”。
- 失败状态展示简短原因和“重新同步”按钮。
- 处理中分块使用渐进轮询获取状态；无处理中分块时停止轮询。
- 切换分块时保留搜索条件、筛选条件和大纲滚动位置。

## 10. 安全与校验

- 所有分块接口必须校验 `agentId`、`knowledgeId` 和 `chunkId` 的归属关系。
- 分块名称去除首尾空白并限制长度。
- 正文不能为空，并设置合理的最大长度。
- Markdown 预览必须进行安全过滤，禁止执行脚本。
- 错误信息返回用户可理解的摘要，服务端日志保留完整异常。

## 11. 测试范围

### 后端

- 分块保存成功并递增版本。
- 乐观锁冲突不会覆盖新内容。
- 连续保存时旧消息不会覆盖最新状态。
- 重复消息消费保持幂等。
- 新向量成功后才删除旧向量。
- 向量化失败自动重试并在死信后标记失败。
- 手动重试仅处理最新版本。
- 名称锁定后正文修改不覆盖名称。
- AI 名称生成旧版本结果不会覆盖最新名称。
- AI 名称生成期间用户手动锁定名称后，晚到结果不会覆盖用户名称。
- 接口归属校验有效。

### 前端

- 文件行可进入分块工作台并正确返回。
- 搜索、状态筛选和分页行为正确。
- 未保存内容切换确认有效。
- 编辑与预览切换正确。
- 保存后展示异步状态而非直接显示已同步。
- 版本冲突时保留本地编辑内容。
- 最终失败时可手动重试。
- 窄屏下左侧大纲可收起，编辑区域不产生无意义横向滚动。

## 12. 实施边界

本设计只覆盖：

- 文件列表到分块工作台的页面流转
- 分块大纲、详情、编辑和预览
- 分块名称生成与锁定
- 单分块版本控制
- 单分块 RocketMQ 重新向量化

本设计明确不覆盖：

- 现有 `agent_knowledge_job` 表
- 现有知识任务调度器
- 现有本地事件机制
- 新增、删除、拆分、合并、排序和批量维护分块
- Outbox 或后台消息补偿扫描
